DATA FLOW
═════════════════════════════════════════════════════════════════════════════

GENERATE BUTTON FLOW:
  User Input (Sidebar)
    ↓
  Validate
    ↓
  DesignBuilder.build()
  - Calculate currents
  - Select MCCBs
  - Size busbar
    ↓
  DesignObject (complete design)
    ↓
  SLDRenderer.render() → SVG
    ↓
  svg_to_png() → PNG (temp)
    ↓
  Store in session_state
    ↓
  Display SLD + metrics on screen

DOWNLOAD BUTTON FLOW:
  Reuse stored DesignObject
    ↓
  GARenderer.render() → SVG
    ↓
  svg_to_png(ga) → PNG (temp)
    ↓
  PDFReportGenerator.generate_full_report()
  - Page 1: Title + Summary
  - Page 2: SLD (image)
  - Page 3: GA (image)
  - Page 4: BOM (table)
    ↓
  PDF bytes in buffer
    ↓
  Cleanup temp files
    ↓
  Download PDF


DESIGN PRINCIPLES
═════════════════════════════════════════════════════════════════════════════

✅ SEPARATION OF CONCERNS
   - Models:   Data structures (no logic)
   - Engine:   Calculations (no UI/rendering)
   - Renders:  Display only (no calculations)
   - UI:       Input/output only (no business logic)

✅ SINGLE RESPONSIBILITY
   - Each module has ONE clear purpose
   - Easy to test, extend, modify

✅ REUSABILITY
   - DesignObject used by all renderers
   - No duplicate calculations
   - Renderers independent of each other

✅ TESTABILITY
   - Pure functions for calculations
   - Mock-friendly architecture
   - Clear input/output contracts

✅ MAINTAINABILITY
   - Clear naming conventions
   - Comprehensive docstrings
   - Type hints throughout
   - Constants centralized
